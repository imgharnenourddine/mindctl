## 🖥️ Partie 1 — Setup WSL

### ⚙️ Étape 1 — Mettre à jour le système

```bash
sudo apt update && sudo apt upgrade -y
```

|Commande|Rôle|
|---|---|
|`sudo`|Exécuter en tant qu'administrateur|
|`apt update`|Télécharger la liste des dernières versions disponibles|
|`&&`|Si la commande précédente réussit → exécuter la suivante|
|`apt upgrade -y`|Installer les mises à jour. `-y` accepte automatiquement|

> [!info] 💡 `apt update` ne met rien à jour — il regarde seulement ce qui existe. C'est `apt upgrade` qui installe vraiment.

---

### 📦 Étape 2 — Installer les dépendances

```bash
sudo apt install -y \
    curl \
    gcc \
    jq \
    make \
    git \
    docker.io \
    postgresql-client \
    mysql-client
```

> [!tip] Le `\` en fin de ligne permet de continuer la commande sur la ligne suivante — c'est juste pour la lisibilité. C'est une seule commande.

|Programme|Rôle dans mindctl|
|---|---|
|`curl`|Appeler l'API Mistral AI et envoyer des requêtes HTTP|
|`gcc`|Compiler les fichiers C — `fork_agent.c` et `thread_agent.c`|
|`jq`|Lire et extraire des données depuis les réponses JSON de l'API|
|`make`|Automatiser la compilation des fichiers C|
|`git`|Gérer le code source et partager avec l'équipe via GitHub|
|`docker.io`|Lancer les projets Docker que depguard va analyser|
|`postgresql-client`|Connecter cleaner et analyzer à une base PostgreSQL|
|`mysql-client`|Connecter cleaner et analyzer à une base MySQL|

### 🗂️ Étape 3 — Créer la structure du projet

```bash
mkdir -p ~/mindctl/{agents,core,helpers,exemple/{test_leger,test_moyen,test_lourd}}
cd ~/mindctl
touch mindctl.sh
touch mindctl.conf
touch agents/{depguard,cleaner,analyzer,transformer,validator,insight}.sh
touch core/{logger,llm,errors}.sh
touch helpers/{fork_agent,thread_agent}.c
chmod +x mindctl.sh agents/*.sh core/*.sh
```

#### 🔍 Explication de chaque commande

**`mkdir -p`**

```
mkdir  → créer un dossier
-p     → deux choses :
         1. créer tous les dossiers parents manquants en cascade
         2. ne pas planter si le dossier existe déjà
```

**`{agents,core,helpers,...}`**

Syntaxe Bash pour créer plusieurs dossiers en une seule commande. Équivalent à :

```bash
mkdir ~/mindctl
mkdir ~/mindctl/agents
mkdir ~/mindctl/core
mkdir ~/mindctl/helpers
mkdir ~/mindctl/exemple
mkdir ~/mindctl/exemple/test_leger
mkdir ~/mindctl/exemple/test_moyen
mkdir ~/mindctl/exemple/test_lourd
```

**`touch`**

```
Crée un fichier vide.
Si le fichier existe déjà → ne fait rien.
Sert à préparer tous les fichiers que chaque membre va remplir.
```

**`chmod +x`**

```
chmod  → changer les permissions d'un fichier
+x     → ajouter la permission d'exécution
*.sh   → tous les fichiers .sh dans ce dossier

Sans ça → Permission denied quand vous lancez ./mindctl.sh
```

> [!warning] ⚠️ Ne pas oublier `chmod +x` — sans ça Linux refuse de lancer vos scripts.

### ⚙️ Étape 4 — Configurer mindctl.conf

```bash
cat > ~/mindctl/mindctl.conf << EOF
MINDCTL_MODEL=mistral-small-latest
MINDCTL_LOG_DIR=/var/log/mindctl
MISTRAL_API_KEY=votre_clé_api_ici
LLM_URL=https://api.mistral.ai/v1/chat/completions
EOF
```

#### 🔍 Explication

| Élément  | Rôle                                                   |
| -------- | ------------------------------------------------------ |
| `cat >`  | Écrire dans un fichier. `>` écrase le contenu existant |
| `<< EOF` | Début d'un bloc de texte multiligne                    |
| `EOF`    | End Of File — marque la fin du bloc                    |

Ce que ça écrit dans `mindctl.conf` :

```
MINDCTL_MODEL=mistral-small-latest    ← modèle Mistral à utiliser
MINDCTL_LOG_DIR=/var/log/mindctl      ← où stocker les logs
MISTRAL_API_KEY=votre_clé_api_ici    ← votre clé API à remplacer
LLM_URL=https://api.mistral.ai/...   ← adresse de l'API Mistral
```

> [!tip] 🔑 Pour obtenir votre clé API gratuite :
> 
> ```
> → console.mistral.ai
> → Sign Up (compte gratuit)
> → API Keys → Create new key
> → Copier la clé dans mindctl.conf
> ```

### 📝 Étape 5 — Créer le dossier de logs

```bash
sudo mkdir -p /var/log/mindctl/reports
sudo chown $USER:$USER /var/log/mindctl
```

#### 🔍 Explication

**Pourquoi `/var/log/` ?**

```
/var/log/ est le dossier standard Linux pour tous les logs système.
Vous n'avez pas le droit d'y écrire sans sudo
car il appartient à root.
```

**`chown $USER:$USER`**

```
chown  → changer le propriétaire d'un dossier
format → chown proprietaire:groupe dossier

$USER:$USER
  │      │
  │      └── groupe       = votre nom d'utilisateur
  └───────── propriétaire = votre nom d'utilisateur

Exemple si vous vous appelez ahmed :
→ chown ahmed:ahmed /var/log/mindctl
```

> [!info] 💡 Pourquoi deux fois `$USER` ? Parce que `chown` a besoin du propriétaire ET du groupe. Sous Linux chaque utilisateur a un groupe qui porte le même nom que lui par défaut — donc les deux sont identiques.

> [!warning] ⚠️ Sans `chown`, quand mindctl essaie d'écrire dans `/var/log/mindctl` vous obtenez `Permission denied`.

## 📄 Partie 3 — Rôle de chaque fichier

### 🎯 Les fichiers racine

#### 📄 `mindctl.sh` — Le chef d'orchestre

```
C'est le seul fichier que l'utilisateur appelle directement.

Il reçoit les options (-a, -p, -f, -t, -s...)
         ↓
Vérifie les paramètres
         ↓
Appelle le bon agent selon ce que l'utilisateur demande

Sans ce fichier → rien ne fonctionne
```

#### ⚙️ `mindctl.conf` — La configuration

```
Contient les variables globales :
→ clé API Mistral
→ URL de l'API
→ dossier des logs
→ modèle LLM à utiliser

Tous les scripts lisent ce fichier au démarrage.
```

---

### 🧠 Dossier `core/` — Les fonctions partagées

> [!warning] ⚠️ Ces 3 fichiers doivent être créés **en premier** car tous les autres scripts en dépendent.

#### `core/logger.sh` — Le système de logs

```
Contient uniquement la fonction log_message()

Chaque fois qu'un agent veut afficher un message :
→ affiche dans le terminal
→ écrit dans /var/log/mindctl/history.log

Format imposé par le prof :
yyyy-mm-dd-hh-mm-ss : username : INFOS : message
yyyy-mm-dd-hh-mm-ss : username : ERROR : message
```

#### `core/llm.sh` — Le cerveau

```
Contient uniquement la fonction appeler_llm()

Chaque fois qu'un agent a besoin d'intelligence :
→ reçoit un prompt en paramètre
→ envoie à l'API Mistral AI via curl
→ retourne la réponse en texte

Utilisé par : depguard, analyzer, insight
```

#### `core/errors.sh` — La gestion des erreurs

```
Contient la fonction erreur() avec 8 codes :

100 → Option inexistante
101 → Paramètre -p manquant
102 → Fichier introuvable
103 → Permission refusée
104 → Format non supporté
105 → API inaccessible
106 → Agent inconnu
107 → Échec du traitement

Après chaque erreur → affiche l'aide automatiquement
```

---

### 🤖 Dossier `agents/` — Les agents spécialisés

#### 🛡️ `agents/depguard.sh` — Le cycle DevOps

```
L'agent le plus important du projet.

Étape 1 → détecte les conflits entre fichiers de config
           Dockerfile vs docker-compose vs package.json...
           Bash détecte, LLM explique et propose la solution

Étape 2 → vérifie que le système Linux est compatible
           node installé ? bonne version ? ports libres ?

Étape 3 → lance le projet et diagnostique les échecs
           capture les logs, vérifie que les services répondent
           si échec → LLM identifie la cause + propose correction

Génère depguard.lock après succès
```

#### 🧹 `agents/cleaner.sh` — Le nettoyeur

```
Lit le contexte transmis par depguard
         ↓
Se connecte directement à la base de données
         ↓
Extrait les données en CSV
         ↓
Supprime :
→ les doublons      (sort | uniq)
→ les lignes vides  (grep -v '^$')
→ les valeurs nulles (awk)
```

#### 📈 `agents/analyzer.sh` — L'analyste

```
Se connecte à la base via le contexte depguard
         ↓
Calcule les statistiques :
→ total de lignes
→ pourcentage de valeurs nulles
→ moyennes et anomalies
         ↓
Envoie à appeler_llm() pour interprétation
         ↓
Affiche les recommandations du LLM
```

#### 🔄 `agents/transformer.sh` — Le convertisseur

```
Transforme les données d'un format à un autre :
→ CSV vers JSON
→ JSON vers CSV

Utilise awk et sed pour les conversions
```

#### ✅ `agents/validator.sh` — Le validateur

```
Vérifie que la structure des données est correcte :
→ nombre de colonnes cohérent sur toutes les lignes
→ formats valides (emails, dates, nombres)
→ aucune valeur hors format

Utilise des expressions régulières Bash
```

#### 🔗 `agents/insight.sh` — Le corrélateur

```
L'agent le plus unique du projet.

Lit les résultats de depguard
         +
Lit les résultats des agents données
         ↓
Envoie les deux ensemble à appeler_llm()
         ↓
LLM trouve les liens entre :
→ anomalies de configuration
→ anomalies dans les données de production

Exemple :
erreur disque à 09h15 dans les logs
= anomalie dans la table orders à 09h15
→ LLM : "la surcharge disque a corrompu des lignes"
```

### ⚡ Dossier `helpers/` — Les modules C

> [!info] 💡 Pourquoi du C ? `fork()` et `pthreads` sont des appels système Linux accessibles uniquement via C. Bash ne peut pas les implémenter nativement.

#### `helpers/fork_agent.c` — Gestionnaire de processus

```
Utilise fork() pour lancer chaque agent
dans un processus fils indépendant.

Si un agent plante → les autres continuent.

Utilisé avec l'option : -f
Compilation : gcc fork_agent.c -o fork_agent
```

```
Processus père (mindctl)
    ├── fork() → Agent table1 → résultat
    ├── fork() → Agent table2 → résultat
    └── fork() → Agent table3 → résultat
              (tous en parallèle)
```

#### `helpers/thread_agent.c` — Gestionnaire de threads

```
Utilise pthreads pour analyser plusieurs tables
simultanément dans le même processus.

Plus léger que fork → idéal pour grandes quantités.

Utilisé avec l'option : -t
Compilation : gcc thread_agent.c -o thread_agent -lpthread
```

```
Processus unique (mindctl)
    ├── Thread 1 → table1 → analyse
    ├── Thread 2 → table2 → analyse
    ├── Thread 3 → table3 → analyse
    └── Thread 4 → table4 → analyse
              (tous simultanément)
```

|            | Fork `-f`             | Threads `-t` |
| ---------- | --------------------- | ------------ |
| Mémoire    | Séparée par processus | Partagée     |
| Isolation  | Totale                | Partielle    |
| Ressources | Plus lourdes          | Plus légères |
| Usage      | 5-10 tables           | 50+ tables   |
