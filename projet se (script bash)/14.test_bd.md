#### 🟢 testdb_leger — Scénario Subshell `-s`

```
1 seule table : users

Colonnes :
→ id, nom, email, age, salaire

Données : 50 lignes avec :
→ doublons      (Ali répété 3 fois)
→ emails nuls   (Nadia sans email)
→ emails invalides (omar_invalid)
→ ages nuls
→ salaires nuls

But du test :
→ cleaner détecte et supprime les doublons
→ cleaner remplace les nulls par la moyenne
→ validator détecte les emails invalides
→ analyzer calcule les stats de la table
→ Mode subshell ( ) — 1 agent, léger
```

#### 🟡 testdb_moyen — Scénario Fork `-f`

```
10 tables :
→ clients, produits, commandes, paiements
→ employes, departements, fournisseurs
→ livraisons, avis, logs_systeme

Données : 5 à 10 lignes par table avec :
→ doublons dans chaque table
→ emails invalides (bad_email)
→ valeurs nulles (prix, stock, salaire...)
→ valeurs négatives (stock = -5)
→ dates nulles

But du test :
→ fork_agent lance 10 processus fils en parallèle
→ chaque fils traite une table indépendamment
→ PID différents visibles dans l'affichage
→ si un fils plante → les autres continuent
→ Mode fork — moyen, isolation totale
```

#### 🔴 testdb_lourd — Scénario Threads `-t` + Pipeline `-w`

```
50 tables identiques :
→ table_01 jusqu'à table_50

Colonnes identiques dans chaque table :
→ id, nom, email, valeur, categorie, date_col, statut

Données : 110 lignes par table avec :
→ 100 lignes normales (nulls + emails invalides)
→ 10 doublons exacts par table
→ valeurs nulles aléatoires (1 ligne sur 7)
→ dates nulles (1 ligne sur 5)

But du test :
→ thread_agent lance 50 threads simultanément
→ mutex protège les affichages
→ pipeline complet :
   depguard → cleaner → analyzer → insight
→ LLM corrèle les anomalies config + données
→ Mode thread — lourd, parallélisme maximal
```
